# FigUI3 UI setup (PropsKit in plugins)

Every GenTool panel uses **PropsKit** controls. In this workspace, PropsKit is implemented with **[FigUI3](https://rog.ie/figui3)** (`@rogieking/figui3`) — web components that match Figma's native properties panel.

The canonical live reference is the **[FigUI3 Playground](https://rog.ie/figui3)**. For panel layout and spacing, match the **[Generative tools effects panel](https://staging.figma.com/design/ChG6473qlBtfrv8VYx1jaR/Generative-tools-effects?node-id=17850-75051)** (240px wide, tight section headers, 32px field rows).

The working UI shell is `scaffold/src/ui.template.html`. Start from that scaffold each run, then use this doc only when you need the rationale or a troubleshooting detail.

---

## File workflow (read this first)

| File | Who edits it | Purpose |
|---|---|---|
| `template/src/ui.template.html` | **You (the agent)** | Source UI: markup, panel spacing overrides, app `<script>` |
| `template/src/ui.html` | **Build output only** | Generated by `npm run bundle-ui`. Figma loads this file. |
| `template/scripts/bundle-ui.mjs` | Pre-configured | Inlines FigUI3 CSS + JS into `ui.html` |
| `template/src/vendor/fig.css`, `fig.js` | Copied on `npm install` | Vendor copies from `@rogieking/figui3` |

**Always edit `ui.template.html`, then run `npm run bundle-ui` (or `npm run build`).** Never hand-edit `ui.html` — your changes will be overwritten.

`code.ts` is edited separately and compiles to `code.js` via `tsc`.

## Why controls silently fail (and how to fix it)

We hit three separate failures getting FigUI3 to render inside Figma's plugin iframe. All three must be addressed.

### 1. Figma only loads `ui.html` — no external assets

The plugin UI is a single iframe pointed at `manifest.json` → `"ui": "ui.html"`. **Linked `<script src="./vendor/fig.js">` or `<link href="./vendor/fig.css">` are ignored or fail silently.** The entire FigUI3 library must be **inlined** into `ui.html`.

**Fix:** The template's `bundle-ui.mjs` replaces `<!-- FIGUI3_CSS -->` and `<!-- FIGUI3_JS -->` placeholders in `ui.template.html` with the full contents of `fig.css` and `fig.js`.

### 2. FigUI3 is a classic script bundle, not an ES module

`fig.js` is an IIFE that registers custom elements on `window`. **`import` from `fig.js` fails silently** in the plugin iframe.

**Fix:** Load with a plain `<script>` tag (after the inlined FigUI3 block), not `<script type="module">`.

```html
<script>
<!-- FIGUI3_JS -->
</script>
<script>
  // Your app logic here
</script>
```

### 3. Init before custom elements are defined

If your app script runs before FigUI3 registers `<fig-dropdown>`, `<fig-field>`, etc., the browser treats them as unknown elements — **empty rows, no inputs, no errors in the main Figma console**.

**Fix:** Wait for every component you use, then init:

```js
Promise.all([
  customElements.whenDefined('fig-group'),
  customElements.whenDefined('fig-header'),
  customElements.whenDefined('fig-footer'),
  customElements.whenDefined('fig-field'),
  customElements.whenDefined('fig-input-number'),
  customElements.whenDefined('fig-slider'),
  customElements.whenDefined('fig-dropdown'),
  customElements.whenDefined('fig-input-palette'),
  customElements.whenDefined('fig-button'),
  // add any other fig-* tags you use
]).then(init).catch(function (err) {
  console.error('FigUI3 failed to load:', err);
});
```

**Debugging tip:** Right-click inside the plugin panel → **Inspect** to open the iframe DevTools. The main Figma app console will not show UI errors.

### 5. Only wait on components that exist in the bundle

The FigUI3 README lists some tags that are **not registered** in `dist/fig.js` (notably `<fig-field-slider>`). If `customElements.whenDefined('fig-field-slider')` is in your `Promise.all`, **init never runs** — no auto-generate, no button handlers, silent failure.

Use `<fig-field>` + `<fig-slider>` for slider rows. Verify tags with:

```bash
node -e "const j=require('fs').readFileSync('src/vendor/fig.js','utf8'); console.log(j.includes('customElements.define(\"fig-slider\")'));"
```

### 6. Bundle script `$&` pitfall

Minified `fig.js` contains `$&` sequences. If `bundle-ui.mjs` uses a string replacer (not a function), those become corrupted and FigUI3 breaks mysteriously.

**Fix (already in template):** use `() => js` and `() => css` as replacer functions in `.replace()`.

## Panel layout (required structure)

Match the Generative tools native pattern. **All visible UI must live inside `<div id="plugin-root">`** — the auto-resize ResizeObserver targets this element. Without it, the panel cannot shrink-to-fit.

```html
<body>

  <div id="plugin-root">

    <fig-group>
      <fig-header borderless><h3>Section name</h3></fig-header>

      <fig-field direction="horizontal" columns="thirds">
        <label for="myControl">Label</label>
        <fig-dropdown id="myControl" value="a">
          <option value="a">Option A</option>
        </fig-dropdown>
      </fig-field>

    <fig-field direction="horizontal" columns="thirds">
      <label for="density">Density</label>
      <fig-slider id="density" variant="neue" value="50" min="0" max="100" text="true" units="%" full></fig-slider>
    </fig-field>
    </fig-group>

    <fig-group>
      <fig-header borderless><h3>Another section</h3></fig-header>
      <!-- more fields -->
    </fig-group>

  <fig-footer borderless>
    <div class="footer-actions">
      <fig-button id="run" variant="primary">Generate grid</fig-button>
    </div>
  </fig-footer>

  </div>

</body>
```

Rules:

- **`<div id="plugin-root">`** wraps every visible element. This is the resize target.
- **240px panel width** in `figma.showUI` (see below).
- **`<fig-group>`** for each logical section. Adjacent groups get an inset top border automatically (with our spacing overrides).
- **`<fig-header borderless><h3>…</h3></fig-header>`** — sentence-case section titles, not uppercase.
- **`<fig-field direction="horizontal" columns="thirds">`** for label + control rows (label left, control right).
- **`<fig-slider variant="neue">`** inside `<fig-field>` for slider rows. Always set `variant="neue"` — it's the PropsKit / GenTool style. There is no `<fig-field-slider>` in the bundled library — waiting on it hangs init forever.
- **`<fig-footer borderless>`** — see **Footer spec** below. Always wrap buttons in `<div class="footer-actions">`. **Generators:** primary **Generate** button only unless the prompt explicitly asks for an additional secondary action. **Actions:** single **Apply** button. **Never auto-post `generate` on plugin open.**

Do **not** add `body { padding: … }`. The panel should be edge-to-edge like Figma's native panels.

## Panel spacing overrides (copy every time)

FigUI3 defaults are looser than Figma's GenTool panels (`fig-header` is 40px tall, groups have 12px margins). **Copy this entire block** into the second `<style>` section of every `ui.template.html`, after the `<!-- FIGUI3_CSS -->` block. It includes **panel spacing, footer spec, and color-picker popover fixes** (required whenever the tool uses `fig-input-color`).

## Color picker (`fig-input-color`)

Use this whenever the tool has a color control. The scaffold ships the CSS block above and the resize JS below — **preserve both** when overwriting `ui.template.html`. Do not reimplement with ad-hoc rules.

### Required markup

FigUI3 defaults are wrong for GenTools. Always set all three attributes:

```html
<fig-field direction="horizontal" columns="thirds">
  <label for="accent">Accent</label>
  <fig-input-color
    id="accent"
    value="#0D99FF"
    text="true"
    alpha="true"
    picker="figma"
  ></fig-input-color>
</fig-field>
```

| Attribute | Default (wrong) | Required |
|---|---|---|
| `text` | `false` | `true` — hex field in panel row |
| `alpha` | `false` | `true` — opacity % + opacity slider in popover |
| `picker` | `native` | `figma` — full Figma-style popover |

Also add `fig-input-color`, `fig-fill-picker`, and `fig-chit` to your `customElements.whenDefined` list, and bind **`change`** (not `input`) for regenerate.

### Why the CSS overrides exist

In a **240px** plugin iframe, three FigUI3 defaults break the popover:

1. **`dialog { max-width: 50vw }`** — caps the picker at 120px; slider rails collapse.
2. **`fig-field` two-control grid (1fr + 2fr)** — footer Hex/HSB row misaligns vs the slider column above.
3. **No rule on `.fig-fill-picker-input-mode`** — if you force `width: 100%` or `min-width: 4.5rem` on the mode dropdown, it overlaps the hex/HSB inputs.

The spacing block fixes all three by:

- Pinning `dialog.fig-fill-picker-dialog` to 240px and allowing full height.
- Retuning `--fig-field-*` ratios on `.fig-fill-picker-inputs` to match `.fig-fill-picker-sliders` (72px eyedropper column + `1fr` track).
- Sizing the mode dropdown to **`3.5rem`** (eyedropper width), centered — not full column width.

### Required resize JS (popover extends below `#plugin-root`)

The color popover is a `<dialog>` appended to `document.body`, outside `#plugin-root`. A ResizeObserver on `#plugin-root` alone clips the bottom. The scaffold ships this pattern — keep it:

```js
function measurePanelHeight() {
  var root = document.getElementById('plugin-root');
  if (!root) return 0;
  var height = Math.ceil(root.getBoundingClientRect().height);
  var dialog = document.querySelector('dialog.fig-fill-picker-dialog[open]');
  if (dialog) {
    var dialogBottom = Math.ceil(dialog.getBoundingClientRect().bottom + 12);
    height = Math.max(height, dialogBottom);
  }
  return height;
}

// Call measurePanelHeight from your ResizeObserver rAF callback, and add a
// MutationObserver on document.body for dialog [open] attribute changes.
```

See the scaffold `<script>` block for the full `watchColorPickerDialog` wiring.

### Do NOT

- Remove the `dialog.fig-fill-picker-dialog { … }` block when the tool has colors.
- Add `min-width` on `.fig-fill-picker-input-mode` (causes overlap with hex/HSB fields).
- Set `width: 100%` on the mode dropdown (stretches past eyedropper into the input column).
- Replace the footer grid with a custom `display: grid !important` — use the `--fig-field-*` variable approach in the spacing block instead.
- Omit `text="true" alpha="true" picker="figma"` — produces a broken minimal swatch.

### Color picker checklist

- [ ] Markup uses `text="true" alpha="true" picker="figma"`
- [ ] Full color-picker CSS block present (from spacing overrides above)
- [ ] `measurePanelHeight` + `watchColorPickerDialog` present in UI script
- [ ] `fig-input-color` in `customElements.whenDefined` list
- [ ] Popover sliders render at full width; Hex/HSB footer aligns with hue slider

### Footer spec

Every Generator (and Action) ships a footer with this exact shape. The point: chrome around the buttons must be identical whether there's 1, 2, or N buttons. Inconsistent footer padding is one of the loudest "this tool was hand-tuned" signals.

| Edge | Spacing |
|---|---|
| Sides (left + right) | **16px** |
| Top (below divider line) | **12px** |
| Bottom (above panel edge) | **12px** |
| Between buttons | **8px** |
| Divider above footer | **1px**, `--figma-color-border` |

The footer gets 12px top + bottom — slightly more than the 8px section-group break rhythm — so the primary action(s) feel like a deliberate destination rather than just another row.

```html
<fig-footer borderless>
  <div class="footer-actions">
    <fig-button id="run" variant="primary">Generate</fig-button>
  </div>
</fig-footer>
```

The CSS (already in the spacing block above):

```css
fig-footer,
fig-footer[borderless] {
  box-shadow: inset 0 1px 0 var(--figma-color-border);
  flex-direction: column;
  align-items: stretch;
  height: auto;
  min-height: 0;
  padding: 12px 16px;
}

fig-footer::before {
  display: none;
}

.footer-actions {
  display: flex;
  flex-direction: column;
  gap: 8px;
}

.footer-actions fig-button {
  width: 100%;
  margin: 0;
}
```

**Why an inner `.footer-actions` stack and not button margins?** Margins force you to special-case the last child (no trailing margin) and the layout breaks the moment you add/remove a button. `flex` with `gap` puts the spacing between buttons, not on them — so a 1-button footer and a 5-button footer both have identical 12px top + 12px bottom + 16px sides + 8px between.

**Generator footer:** primary **Generate** only (default). Secondary actions (Reshuffle, Reset, …) **only when the user's prompt explicitly requests one and describes what it does**.
**Action footer:** single primary `Apply` — still wrap it in `.footer-actions` for consistency.

Hardcoded pixel values (12 / 16 / 8) on purpose: the footer is a fixed visual element and tying it to literal pixels keeps it stable if the spacer scale ever shifts. Don't change these unless the entire spec changes.

### Why `flex-direction: column` on headers

`fig-header` is a **horizontal flex row** in FigUI3. Never put `::before` pseudo-elements inside it for separators — they layout **beside** the title, squishing section names off-screen. Section dividers belong on `fig-group + fig-group` (inset border + padding), not inside the header.

### Spacing tokens (from Figma reference)

| Element | Spec |
|---|---|
| Panel width | 240px |
| Field row | 32px (`py-4`, `px-16`) — handled by `fig-field` tokens |
| First section title | 12px below panel top (group `padding-top: 12px` + header `padding-top: --spacer-1`) |
| Between sections | 8px above divider + 1px line + 8px below divider |
| Footer | 1px divider + 12px top + button(s) at 8px gaps + 12px bottom, 16px sides |

## `figma.showUI` dimensions

```ts
figma.showUI(__html__, { width: 240, height: 320, themeColors: true });
```

- **`width: 240`** — matches GenTool panel spec.

- **`themeColors: true`** — required. FigUI3 reads `--figma-color-*` and `--spacer-*` tokens.

## Auto-resize: hug content always

**Every GenTool panel must shrink-to-fit its content** — both on initial paint and whenever a collapsible row opens or closes. The Figma plugin iframe does not auto-size to its HTML document; you have to measure inside the UI and call `figma.ui.resize(width, height)` in the sandbox.

### The architecture

A Figma plugin has two separate environments:

- **Sandbox (`code.ts`)** — runs your core logic, has access to the Figma node tree, no DOM access.
- **Iframe UI (`ui.html`)** — standard webpage with DOM + browser APIs, no canvas access.

Changing an element's height inside the document does **not** automatically resize the iframe. You must measure in the UI, post a message across the boundary, and call `figma.ui.resize` in the sandbox.

### Three ingredients

**1. Override the vendor `height: 100%`.** FigUI3's `fig.css` sets `html { height: 100% }` and `body` inherits 100%. This pins the document to the iframe's current height — a ResizeObserver on the body will never see it shrink. **Copy these rules into every tool's panel CSS:**

```css
html, body {
  margin: 0;
  padding: 0;
  height: auto;
  min-height: 0;
}

#plugin-root {
  height: auto;
}
```

The panel spacing CSS block in the section above already includes these. Do not omit them.

**2. Wrap your UI in a single `#plugin-root` div, and observe it.** In `ui.template.html`:

```html
<body>
  <div id="plugin-root">
    <fig-group> … </fig-group>
    <fig-footer borderless> … </fig-footer>
  </div>

  <script><!-- FIGUI3_JS --></script>
  <script>
    var lastReportedHeight = 0;
    var resizeRaf = 0;

    function reportHeight(height) {
      var h = Math.ceil(height);
      if (h && h !== lastReportedHeight) {
        lastReportedHeight = h;
        parent.postMessage({ pluginMessage: { type: 'resize', height: h } }, '*');
      }
    }

    (function setupAutoResize() {
      var root = document.getElementById('plugin-root');
      if (!root || typeof ResizeObserver === 'undefined') return;
      var observer = new ResizeObserver(function (entries) {
        // rAF-debounce — CSS transitions fire one observation per frame.
        if (resizeRaf) cancelAnimationFrame(resizeRaf);
        resizeRaf = requestAnimationFrame(function () {
          resizeRaf = 0;
          for (var i = 0; i < entries.length; i++) {
            reportHeight(entries[i].contentRect.height);
          }
        });
      });
      observer.observe(root);
    })();
  </script>
</body>
```

The scaffold ships this exact wiring. `entry.contentRect.height` includes children's margins (it's the content-box of `#plugin-root` which contains the fig-groups), shrinks when content collapses, and grows when content expands.

**3. Handle the resize message in the sandbox.** In `code.ts`:

```ts
type UiToCodeMessage =
  | { type: 'generate'; state: State }
  | { type: 'regenerate'; state: State }
  | { type: 'resize'; height: number };

figma.ui.onmessage = async (msg: UiToCodeMessage) => {
  if (msg.type === 'resize') {
    const h = Math.max(120, Math.min(900, Math.round(msg.height)));
    figma.ui.resize(240, h);
    return;
  }
  // ... other handlers ...
};
```

Clamp aggressively (`120..900`) so a broken measurement can't make the panel disappear or fill the screen. The scaffold ships this handler.

### Animating collapsible sections

When a row's height changes (palette shown only in "Custom" theme, advanced section toggle, etc.), animate the row with a CSS grid trick — the ResizeObserver on `#plugin-root` follows each frame and the panel glides with it:

```css
.collapsible {
  display: grid;
  grid-template-rows: 1fr;
  transition: grid-template-rows 200ms ease, opacity 160ms ease;
  opacity: 1;
}
.collapsible > .collapsible-inner {
  overflow: hidden;
  min-height: 0;
}
.collapsible[collapsed] {
  grid-template-rows: 0fr;
  opacity: 0;
  pointer-events: none;
}
```

```html
<div class="collapsible" id="advancedRow">
  <div class="collapsible-inner">
    <fig-field direction="horizontal" columns="thirds">
      <label for="advanced">Advanced</label>
      <fig-input-number id="advanced" value="0"></fig-input-number>
    </fig-field>
  </div>
</div>
```

Toggle the `collapsed` attribute in JS. The rAF-debounce in the observer keeps message traffic to one resize per frame.

### Gotchas

- **The 100vh / 100% pitfall.** If `html`, `body`, or your root wrapper has `height: 100%` or `100vh`, the ResizeObserver thinks the element never changes size — because it's bound to the viewport, not the content. Always `height: auto` on the wrapper. (FigUI3's vendor CSS sets `html { height: 100% }`; override it.)
- **`scrollHeight` alone is unreliable.** Once the iframe grows, `document.documentElement.scrollHeight` stays at the iframe size and won't report shrinks. Measure `entry.contentRect.height` from a ResizeObserver on a `height: auto` wrapper instead.
- **`getBoundingClientRect().bottom` ignores trailing margins.** Don't sum child rects; that approach cuts off bottom padding. The observer's `contentRect` reads from the parent's content box, which does account for children's margins.
- **Debounce.** CSS height transitions fire one observation per frame. Wrap the `postMessage` in `requestAnimationFrame` (or coalesce with the previous frame's rAF id) so you send one message per frame, not one per observation.
- **Clamp in the sandbox.** Always bound the height (`Math.max(min, Math.min(max, ...))`) before `figma.ui.resize`, so a measurement bug can't break the panel.

### Checklist

- [ ] UI wrapped in a single `<div id="plugin-root">` (everything visible is inside it)
- [ ] Panel CSS sets `html, body, #plugin-root { height: auto; min-height: 0 }`
- [ ] `code.ts` handles `{ type: 'resize'; height }` and calls `figma.ui.resize(240, clamped)`
- [ ] UI installs a `ResizeObserver` on `#plugin-root`, reads `entry.contentRect.height`, posts via `parent.postMessage`
- [ ] `postMessage` calls are rAF-debounced (one per frame)
- [ ] `figma.showUI` initial height is a sensible splash value (~320); real height arrives on first paint
- [ ] Collapsible rows use `grid-template-rows: 1fr ↔ 0fr` so the height transition is observable
- [ ] If the tool uses `<fig-input-color>`: see **Color picker** section above (`measurePanelHeight`, CSS block, required attrs)

## PropsKit control mapping (FigUI3 web components)

Use these instead of raw HTML + `.propskit-*` classes.

| Need | FigUI3 component | Notes |
|---|---|---|
| Number | `<fig-input-number>` inside `<fig-field>` | `min`, `max`, `steppers="true"` |
| Slider | `<fig-slider variant="neue">` inside `<fig-field>` | Always set `variant="neue"` (PropsKit style). `min`, `max`, `value`, `text="true"`, optional `units` |
| Switch | `<fig-switch>` inside `<fig-field>` | Boolean |
| Dropdown | `<fig-dropdown>` inside `<fig-field>` | `<option value="…">` children |
| Text | `<fig-input-text>` inside `<fig-field>` | Single line |
| Color | `<fig-input-color text="true" alpha="true" picker="figma">` inside `<fig-field>` | **Required attrs** — defaults break the popover. Preserve the color-picker CSS block and resize JS from the scaffold. See **`docs/08-figui3-ui.md > Color picker`**. |
| Palette | `<fig-input-palette>` inside `<fig-field>` | `value='["#hex",…]'` JSON array |
| Primary action | `<fig-button variant="primary">` inside `<div class="footer-actions">` in `<fig-footer>` | Full-width, 1 per footer |
| Secondary action | `<fig-button variant="secondary">` inside `<div class="footer-actions">` in `<fig-footer>` | Full-width via `.footer-actions` stack |

Visit [rog.ie/figui3](https://rog.ie/figui3) for attributes, events, and edge cases.

### Reading / writing values

FigUI3 components expose `.value` but also use the `value` attribute. **Read both; write both** when rehydrating from `loadState`:

```js
function readValue(node) {
  if (!node) return '';
  if (typeof node.value !== 'undefined' && node.value !== null && node.value !== '') {
    return String(node.value);
  }
  return node.getAttribute('value') || '';
}

function writeValue(node, value) {
  if (!node) return;
  node.setAttribute('value', String(value));
  if ('value' in node) node.value = String(value);
}
```

### Commit-fire events

Bind **`change`** on FigUI3 controls (same as PropsKit):

```js
document.querySelectorAll('fig-input-number, fig-slider, fig-dropdown, fig-input-palette').forEach(function (control) {
  control.addEventListener('change', fireRegenerate);
});
```

Sliders inside `fig-slider` commit on release, not drag. Never bind `input`.

## Generator init pattern (no auto-generate)

Replace `DOMContentLoaded` with `customElements.whenDefined` (see above). **Do not send `generate` from `init()`.** Track whether output exists with `hasGenerated`. The auto-resize wiring runs separately (see Auto-resize section above; the scaffold ships it).

```js
var hasGenerated = false;
var outputSelected = false;

function fireRegenerate() {
  if (!hasGenerated || !outputSelected) return;
  parent.postMessage({
    pluginMessage: { type: 'regenerate', state: getStateFromForm() },
  }, '*');
}

Promise.all([
  customElements.whenDefined('fig-group'),
  customElements.whenDefined('fig-field'),
  customElements.whenDefined('fig-dropdown'),
  customElements.whenDefined('fig-slider'),
  customElements.whenDefined('fig-button'),
  // add every other fig-* tag you use
]).then(function () {
  document.getElementById('run').addEventListener('click', function () {
    hasGenerated = true;
    parent.postMessage({
      pluginMessage: { type: 'generate', state: getStateFromForm() },
    }, '*');
  });

  document.querySelectorAll('fig-slider, fig-dropdown, fig-input-number').forEach(function (control) {
    control.addEventListener('change', fireRegenerate);
  });
});

window.onmessage = function (event) {
  var msg = event.data.pluginMessage;
  if (msg && msg.type === 'loadState') {
    setStateInForm(msg.state);
    hasGenerated = true;
  }
  if (msg && msg.type === 'outputSelected') {
    outputSelected = !!msg.selected;
  }
};
```

**Actions:** same FigUI3 setup, but footer posts `apply` on click — no `hasGenerated` gate needed.

**Output targeting:** `generate` always posts `{ type: 'generate' }`. The sandbox decides create vs update. UI `regenerate` only fires when `outputSelected` is true. See `docs/07-plugin-practices.md > Output targeting`.

**Relaunch buttons** (`figma.command === 'regenerate'`) are explicit user actions from the Inspect panel — those may still auto-run in `code.ts`.

### Collapsible rows that animate the panel height

When a control should appear/disappear based on another control's value (e.g. palette only when theme is "custom"), use the `.collapsible` pattern. The ResizeObserver on `#plugin-root` follows the height transition and the panel glides in sync.

```html
<div class="collapsible" id="paletteRow">
  <div class="collapsible-inner">
    <fig-field direction="horizontal" columns="thirds">
      <label for="palette">Palette</label>
      <fig-input-palette id="palette" value='["#000","#fff"]'></fig-input-palette>
    </fig-field>
  </div>
</div>
```

```js
function syncPaletteVisibility(theme) {
  var row = document.getElementById('paletteRow');
  if (!row) return;
  if (theme === 'custom') row.removeAttribute('collapsed');
  else row.setAttribute('collapsed', '');
}

themeDropdown.addEventListener('change', function () {
  syncPaletteVisibility(readValue(themeDropdown));
});
syncPaletteVisibility(readValue(themeDropdown));
```

Toggle `collapsed` on the wrapper; the CSS in the spacing block handles the rest.
